# cryptd — Distribution

How users install the `cryptd` binary.

---

## Context

`cryptd` is the Go backend for the [crypt](https://github.com/punt-labs/crypt)
Claude Code plugin. It is not a standalone user-facing tool — the plugin's
`install.sh` is responsible for getting the binary onto the user's machine. End
users interact with `/crypt` (the plugin skill), not `cryptd` directly.

This differs from the Python projects in the org, which distribute via PyPI
(`uv tool install`). Go projects distribute as pre-built platform binaries.

---

## Distribution Channels

| Channel | Who Uses It | How |
|---------|------------|-----|
| **GitHub Releases** | Plugin `install.sh`, CI, automation | Pre-built binaries per OS/arch with SHA256 checksums |
| **Homebrew Tap** | Power users who prefer `brew install` / `brew upgrade` | Formula auto-generated by GoReleaser into `punt-labs/homebrew-tap` |
| **Build from source** | Contributors building from source | `make build` (or `go build -o cryptd ./cmd/cryptd && go build -o crypt ./cmd/crypt`) |

GitHub Releases is the primary channel. Homebrew and `go install` are
convenience layers — the plugin does not depend on either.

---

## Release Artifacts

Each GitHub Release (triggered by a version tag) produces two binaries per platform:

```text
cryptd_<version>_darwin_arm64.tar.gz   (contains: cryptd, crypt, LICENSE)
cryptd_<version>_darwin_amd64.tar.gz
cryptd_<version>_linux_amd64.tar.gz
cryptd_<version>_linux_arm64.tar.gz
cryptd_<version>_checksums.txt
```

Each tarball contains both binaries (`cryptd` the server and `crypt` the client)
plus a `LICENSE` file. The checksums file contains SHA256 hashes for all archives.

---

## GoReleaser

[GoReleaser](https://goreleaser.com) automates the entire release pipeline.
A `.goreleaser.yaml` in the repo root configures:

1. **Cross-compilation** — `darwin/arm64`, `darwin/amd64`, `linux/amd64`,
   `linux/arm64`
2. **Archives** — tar.gz with binary + LICENSE
3. **Checksums** — SHA256 for all archives
4. **GitHub Release** — uploads archives and checksums, generates changelog
   from conventional commits
5. **Homebrew Tap** — auto-updates the formula in `punt-labs/homebrew-tap`

### Release Flow

```text
git tag v0.3.0
git push origin v0.3.0
    │
    ▼
GitHub Actions (.github/workflows/release.yml)
    │
    ├── goreleaser release
    │       ├── cross-compile (4 targets)
    │       ├── create GitHub Release with archives + checksums
    │       └── update punt-labs/homebrew-tap formula
    │
    └── done
```

### Tag Convention

Tags follow semver: `v0.1.0`, `v0.2.0`, `v1.0.0`. Pre-release tags
(`v0.3.0-rc.1`) produce GitHub pre-releases and do not update Homebrew.

## Plugin Install Script

The `crypt` plugin's `install.sh` handles `cryptd` installation:

1. Detect platform (`uname -s`, `uname -m`)
2. Resolve the download URL for the matching archive from the latest
   GitHub Release (or a pinned version)
3. Download and verify SHA256 checksum
4. Extract the binary to a known location on `$PATH`
   (e.g., `~/.local/bin/cryptd` or `~/.crypt/bin/cryptd`)
5. Verify the binary runs: `cryptd --version` (exit 0)

The install script lives in the `crypt` plugin repo, not here. This repo
produces the artifacts that script consumes.

---

## Trust Tiers

Following the punt-kit distribution standard, the plugin README presents three
install tiers:

| Tier | What the user does | Trust model |
|------|--------------------|-------------|
| **Convenience** | `curl -fsSL .../install.sh \| sh` | Trusts pinned SHA + HTTPS + GitHub |
| **Inspect** | Download, read, then run `sh install.sh` | Reads before executing |
| **Verify** | Download, `shasum -a 256`, compare, then run | Cryptographic verification |

The checksums file in each GitHub Release enables the Verify tier for the
binary itself, independent of the install script.

## Homebrew

A [Homebrew tap](https://docs.brew.sh/Taps) at `punt-labs/homebrew-tap`
hosts the formula. GoReleaser auto-updates it on each release.

```bash
brew install punt-labs/tap/cryptd
brew upgrade cryptd
```

The formula downloads the pre-built binary from GitHub Releases (not building
from source). This keeps install times fast and avoids requiring the Go
toolchain on the user's machine.

---

## Build from Source

For contributors who have the Go toolchain:

```bash
make build
# or individually:
go build -o cryptd ./cmd/cryptd
go build -o crypt ./cmd/crypt
```

This builds both binaries from source. It is not the recommended path for end
users — they should use the plugin installer or Homebrew.

## What This Repo Produces vs. What the Plugin Repo Owns

| Concern | Owned by |
|---------|----------|
| Binary compilation and cross-platform archives | `cryptd` (this repo) |
| GoReleaser config and release CI | `cryptd` (this repo) |
| Homebrew tap formula template | `cryptd` (this repo, via GoReleaser) |
| `install.sh` that downloads and installs | `crypt` (plugin repo) |
| `uninstall.sh` that removes the binary | `crypt` (plugin repo) |
| Plugin marketplace registration | `crypt` (plugin repo) |
| Version pinning for `cryptd` in plugin | `crypt` (plugin repo) |

This separation means `cryptd` can release independently. The plugin pins
a specific `cryptd` version and bumps it when ready.
